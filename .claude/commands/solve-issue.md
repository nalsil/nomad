# GitHub Issue 해결 계획 수립

GitHub 이슈 번호를 입력받아 이슈를 분석하고 상세한 해결 계획을 작성합니다.

## 사용법

```
/solve-issue $ISSUE_NUMBER
```

**예시**:
```
/solve-issue 42
/solve-issue #123
```

## 실행 프로세스

### 1. 이슈 가져오기

먼저 `gh` CLI를 사용하여 이슈를 가져옵니다:

```bash
gh issue view $ISSUE_NUMBER --json title,body,labels,assignees,state,author,createdAt,updatedAt
```

### 2. 이슈 분석

가져온 이슈 정보를 분석합니다:

```markdown
## 📋 이슈 정보

**이슈 번호**: #{NUMBER}
**제목**: {TITLE}
**상태**: {STATE}
**작성자**: {AUTHOR}
**라벨**: {LABELS}
**생성일**: {CREATED_AT}

---

## 📝 이슈 내용 요약

{이슈 본문을 간결하게 요약}

**핵심 문제**:
- {문제점 1}
- {문제점 2}

**요구사항**:
- {요구사항 1}
- {요구사항 2}

**제약사항**:
- {제약사항 1}
- {제약사항 2}
```

### 3. 문제 분석

이슈 타입에 따라 분석을 진행합니다:

#### Bug 이슈 분석
```markdown
## 🔍 문제 분석

### 증상
{보고된 버그 증상}

### 예상 원인
1. **가능성 높음**: {원인 1}
   - 근거: {이유}
   - 영향 범위: {범위}

2. **가능성 중간**: {원인 2}
   - 근거: {이유}
   - 영향 범위: {범위}

### 영향 받는 컴포넌트/파일
- `{파일경로}` - {이유}
- `{파일경로}` - {이유}

### 재현 방법 검증
- [ ] 재현 단계가 명확한가?
- [ ] 로컬 환경에서 재현 가능한가?
- [ ] 추가 정보가 필요한가?
```

#### Feature/Enhancement 이슈 분석
```markdown
## 🔍 요구사항 분석

### 사용자 스토리
As a {사용자},

### 기능 범위
**포함**:
- {포함 항목 1}
- {포함 항목 2}

**제외**:
- {제외 항목 1}
- {제외 항목 2}

### 기술적 고려사항
- {고려사항 1}
- {고려사항 2}

### 의존성
- {의존하는 기능/라이브러리}
- {선행 작업}
```

### 4. 해결 계획 수립

```markdown

---

### Phase 1: 조사 및 분석
**목표**: 문제의 정확한 원인 파악 및 해결 방법 검증

#### 📌 작업 1.1: 코드베이스 분석
- **설명**: 관련 코드 및 의존성 파악

  - `{파일경로}` - {이유}
  - `{파일경로}` - {이유}
- **완료 조건**: 문제 발생 지점 특정
- **예상 시간**: {X분}
- **검증**: {확인 방법}

#### 📌 작업 1.2: 재현 환경 구성
- **설명**: 로컬 환경에서 이슈 재현
- **완료 조건**: 안정적으로 재현 가능
- **예상 시간**: {X분}
- **검증**: {확인 방법}

#### 📌 작업 1.3: 근본 원인 확인
- **설명**: 표면적 증상이 아닌 근본 원인 파악
- **완료 조건**: 왜 발생하는지 명확히 이해
- **예상 시간**: {X분}
- **검증**: {확인 방법}

**Phase 1 완료 기준**:
- [ ] 문제 원인 명확히 파악

### Phase 2: 해결 방법 구현
**목표**: 문제를 해결하는 코드 작성

  ```

#### 📌 작업 2.2: {구체적 작업명}

- **파일**: `{파일경로}`
- **완료 조건**: {측정 가능한 기준}
- **예상 시간**: {X시간}
- **의존성**: 작업 2.1
- **검증**: {확인 방법}

- [ ] 코드 변경 완료

### Phase 3: 테스트 및 검증
**목표**: 수정사항이 의도대로 작동하는지 확인

#### 📌 작업 3.1: 단위 테스트

#### 📌 작업 3.2: 통합 테스트

## 특수 케이스 처리

### 이슈를 찾을 수 없는 경우
```markdown
❌ 이슈 #{NUMBER}를 찾을 수 없습니다.

**확인사항**:
- [ ] 이슈 번호가 정확한가요?
- [ ] 이슈가 삭제되었나요?
- [ ] 저장소 권한이 있나요?

**다시 시도**:
`/solve-issue {올바른 이슈 번호}`
```

### 이미 닫힌 이슈인 경우
```markdown
⚠️ 이슈 #{NUMBER}는 이미 닫힌 상태입니다.

**상태**: {CLOSED/MERGED}
**닫힌 날짜**: {DATE}
**닫은 사람**: {USER}

**진행 방법**:
1. 이슈를 다시 열기: `gh issue reopen {NUMBER}`
2. 새로운 이슈 생성: `/issue {설명}`
3. 그대로 진행: "닫힌 이슈지만 계획을 수립하겠습니다"
```

### gh CLI 미설치/미인증
```markdown
❌ gh CLI를 사용할 수 없습니다.

**설치 확인**:
```bash
gh --version
```

**설치 방법**:
- Windows: `winget install GitHub.cli`
- macOS: `brew install gh`

**인증**:
```bash
gh auth login
```

**대안**:
이슈 내용을 직접 붙여넣어주시면 계획을 수립하겠습니다.
```

## 분석 원칙

### DO ✅
- 이슈 내용을 정확히 이해
- 실행 가능한 작은 단위로 분해
- 각 작업에 명확한 완료 조건
- 리스크를 사전에 식별
- 대안 접근법 고려
- 사용자 피드백 요청

### DON'T ❌
- 이슈 내용을 임의로 해석
- 너무 큰 작업 단위
- 모호한 완료 조건
- 리스크 무시
- 한 가지 방법만 고집
- 사용자 확인 없이 진행

## 사용 예시

### 예시 1: 버그 이슈
```
사용자: /solve-issue 42

시스템:
1. gh issue view 42 실행
2. 이슈 내용 분석
3. 버그 원인 추정
4. 4개 Phase로 해결 계획 수립
5. 사용자에게 검토 요청
```

### 예시 2: 기능 이슈
```
사용자: /solve-issue #123

시스템:
1. gh issue view 123 실행
2. 요구사항 분석
3. 기능 범위 정의
4. 4개 Phase로 구현 계획 수립
5. 사용자에게 검토 요청
```

## 통합 워크플로우

```
/solve-issue 42
  ↓
이슈 분석 및 계획 수립
  ↓
사용자 검토 및 승인
  ↓
브랜치 생성 (git checkout -b issue-42)
  ↓
Phase 1-4 순차 진행
  ↓
각 Phase마다 체크리스트 확인
  ↓
/create-pr (PR 생성)
  ↓
이슈와 자동 연결 (Closes #42)
```
